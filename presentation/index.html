TF Reviewer

An AI agent that reviews your Terraform pull requests with the plan in hand, against your company's standards.

Gabriel, DevOps Engineer

Why

Four problems with Terraform review

01

Review without a plan

An AI assistant reviewing Terraform usually has no plan output. Without it, the review is guesswork about what actually changes.

02

Standards slip through

Approved modules. Required tags. If nobody checks them on every PR, they quietly stop being standards.

03

Drift with no story

Terraform tells you drift exists. It doesn't tell you what changed, when, or who did it.

04

People are the bottleneck

Senior review time is the scarcest resource. It shouldn't be spent on the small stuff.

The idea

Give the agent the plan, the rules,
and real credentials

  • It runs terraform plan itself so the review is based on what will actually change.
  • It reads your dos and don'ts so your standards become the review checklist.
  • It can query AWS so drift gets investigated, not just reported.
  • It comments on the PR in ~2 minutes before a human ever opens it.

Plan and review only. It never applies.

Scope

What it does not do

  • Only what's in the prompt. No cost analysis, no deep security scan, and no best-practice sweep unless you ask for it.
  • Bigger ask, bigger model, bigger bill. Balance cost against the review you actually need.
  • GitHub only. This is built for GitHub Actions. It won't run elsewhere as-is.
  • One environment. No dev / UAT / staging / prod selection in this demo.

Under the hood

Three files. That's the whole thing.

.github/codex/dos-and-donts.md

Your company context, not the prompt. Approved modules, required tags (Project, BillingCode), remote state, no laptop applies.

.github/codex/prompts/tf-plan-review.md

The actual prompt. Read the standards, run the plan, investigate drift, and show the commands you ran so I can reproduce your findings.

.github/workflows/tf-codex-review.yml

71 lines. Credentials → Terraform setup → backend config → init → Codex → comment on the PR.

Secrets

Three secrets, two variables

  • Secrets are OPENAI_API_KEY, AWS_ACCESS_KEY_ID, and AWS_SECRET_ACCESS_KEY. Add kubectl credentials the same way if you want Kubernetes investigation.
  • Variables are AWS_REGION and TF_STATE_BUCKET.
Being honest about this: the agent runs with sandbox: danger-full-access and live AWS keys, and that is why drift investigation works. In production I'd use OIDC with a scoped, read-only role instead of long-lived keys. My own don'ts say exactly that.
Honest limitations

What I'd fix next

  • Publishing every command. Great for reproducibility. But the agent could one day echo a secret into a public PR comment. Rare, but real.
  • I don't agree with every finding. The prompt needs tuning. That's the job, not a bug.
  • Single environment. In production you pick which environment to plan against. This demo doesn't.
  • Static AWS keys. Should be OIDC with a read-only role.
